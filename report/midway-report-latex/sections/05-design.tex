% ШИНЭ ДИАГРАМ ХЭРЭГТЭЙ: Архитектурын диаграмыг шинэчлэх. Дөрвөн давхаргыг тодорхой харуулах:

Эхний давхарга нь Chrome extension-д суурилсан хэрэглэгчийн интерфейс юм. Энэ нь хэрэглэгчийн байгалийн хэлний оролтыг хүлээн авч, AI серверт дамжуулж, хариултыг харуулах үүрэгтэй. WXT framework болон React ашиглан бүтээгдсэн бөгөөд side panel хэлбэрээр ажиллана.

Хоёр дахь давхарга нь AI сервер буюу orchestration давхарга юм. Энэ нь хэлний загвартай харилцах, хэрэглэгчийн хүсэлтийг ойлгох, шаардлагатай MCP хэрэгслүүдийг дуудах, үр дүнг нэгтгэн хариулт үүсгэх үүрэгтэй. Elysia framework болон Vercel AI SDK ашиглан хэрэгжүүлсэн.

Гурав дахь давхарга нь MCP серверүүд юм. Одоогийн байдлаар SISI порталын API-тай холбогдох нэг MCP сервер ажиллаж байна. Энэ нь оюутны мэдээлэл, хуваарь, дүн, санхүү, мэдэгдэл зэрэг домэйнүүдийг хамарсан найман read-only хэрэгсэлтэй. Ирээдүйд багшийн портал, календарь, бусад системүүдийн MCP серверүүд нэмэгдэх боломжтой.

Дөрөв дэх давхарга нь гаднах системүүд юм. Үүнд SISI порталын таван API endpoint (auth, support, tree, phost, teach-sisi), ирээдүйн багшийн порталын API, бусад MCP нийцтэй үйлчилгээнүүд багтана.

Энэхүү давхаргат архитектурын гол давуу тал нь бүрэлдэхүүн хэсгүүдийг тусгаарлаж өгсөнд оршино. Chrome extension, AI сервер, MCP серверүүдийг бие даан хөгжүүлэх, туршихад тохиромжтой. Шинэ MCP сервер нэмэхэд AI серверийн үндсэн логикийг өөрчлөх шаардлагагүй, зөвхөн шинэ хэрэгслийг бүртгэхэд хангалттай.

\section{Модулийн бүтэц}

Одоогийн кодын сан нь monorepo бүтэцтэй бөгөөд Turborepo ашиглан удирддаг. Модулиудыг apps болон packages гэсэн хоёр хэсэгт хуваасан.

\subsection{Applications (apps)}

\texttt{apps/extension} нь Chrome extension-ийн үндсэн код юм. WXT framework ашиглан бүтээгдсэн бөгөөд React 19-д суурилсан хэрэглэгчийн интерфейстэй. Side panel архитектуртай буюу extension-ийн icon дээр дарахад хөтчийн хажуу талд панел нээгдэнэ. Одоогийн байдлаар чат интерфейс бэлэн болсон боловч бодит AI сервертэй холбогдоогүй, mock хариултуудтай ажиллаж байна.

\texttt{apps/server} нь AI серверийн код юм. Elysia framework дээр бүтээгдсэн бөгөөд нэг \texttt{/ai} POST endpoint-тэй. Vercel AI SDK ашиглан Google Gemini загвартай холбогдож, streaming хариулт авах боломжтой. Одоогоор MCP хэрэгслүүдтэй холбогдоогүй, зөвхөн энгийн чат completion ажиллаж байна.

\texttt{apps/web} нь туршилтын зориулалттай вэб интерфейс юм. TanStack Router ашиглан бүтээгдсэн бөгөөд \texttt{/ai} route дээр энгийн чат интерфейстэй. Эцсийн хэрэглээний үндсэн клиент биш боловч хөгжүүлэлтийн явцад прототип турших, debug хийхэд ашиглагдаж байна.

\subsection{Packages}

\texttt{packages/mcp} нь SISI порталд зориулсан локал MCP сервер юм. HTTP сервер хэлбэрээр 3901 порт дээр ажиллана. Найман read-only хэрэгсэл (tool) тодорхойлогдсон бөгөөд session удирдлагын endpoint-уудтай (\texttt{/local/session/register}, \texttt{/local/session/refresh}, \texttt{/local/session/clear}). SISI-ийн таван API endpoint-той холбогдох upstream client хэрэгжүүлсэн.

\texttt{packages/ui} нь хуваалцсан UI компонентуудын сан юм. Shadcn/ui-д суурилсан бөгөөд AI чат элементүүд (Conversation, Message, PromptInput, Reasoning, Sources) болон үндсэн UI компонентууд (Button, Card, Dialog, Input гэх мэт) багтдаг. Tailwind CSS v4 ашигладаг.

\texttt{packages/env} нь environment variable-уудын validation хийдэг сан юм. t3-env ашиглан сервер болон клиент талын environment variable-уудыг шалгадаг.

\texttt{packages/db} нь өгөгдлийн сангийн давхарга юм. Drizzle ORM болон PostgreSQL ашиглахаар тохируулсан. Одоогоор schema тодорхойлогдоогүй буюу хоосон байгаа бөгөөд ирээдүйд чат түүх, audit log, хэрэглэгчийн тохиргоо зэргийг хадгалахад ашиглагдана.

\texttt{packages/infra} нь infrastructure as code юм. Alchemy ашиглан Cloudflare-д deploy хийхээр төлөвлөсөн боловч одоогоор бүрэн хэрэгжээгүй.

\section{Session удирдлага}

SISI API-д хандахын тулд хэрэглэгчийн хүчинтэй сесс шаардлагатай. Иймээс локал MCP сервер нь session удирдлагын тусгай механизмтэй. Энэхүү урсгалыг Зураг~\ref{fig:session-flow}-д үзүүлэв.

\begin{figure}[H]
  \centering
  \includegraphics[width=\textwidth]{assets/diagrams/session-bootstrap.png}
  \caption{Session bootstrap ба MCP ашиглалтын урсгал}
  \label{fig:session-flow}
\end{figure}

% ШИНЭ ДИАГРАМ ХЭРЭГТЭЙ: Session bootstrap урсгалыг sequence диаграмаар харуулах:
% 1. Хэрэглэгч SISI порталд нэвтэрсэн (browser-д cookie байгаа)
% 2. Chrome Extension идэвхжиж, cookie/token авах
% 3. Extension -> MCP Server: POST /local/session/register (cookie/token-той)
% 4. MCP Server: Session store-д хадгалах, idle TTL эхлүүлэх
% 5. MCP Server -> Extension: Session ID буцаах
% 6. Extension -> AI Server: Хэрэглэгчийн асуулт + Session ID
% 7. AI Server -> MCP Server: Tool call (Session ID-тай)
% 8. MCP Server: Session шалгах, SISI API руу хүсэлт илгээх
% 9. MCP Server -> AI Server: Tool result
% 10. AI Server -> Extension: Хариулт

Session bootstrap нь дараах байдлаар ажиллана. Эхлээд хэрэглэгч SISI порталд хөтчөөрөө нэвтэрсэн байна. Chrome extension идэвхжихэд порталын cookie эсвэл access token-ийг авч, MCP серверийн \texttt{/local/session/register} endpoint руу илгээнэ. MCP сервер нь session-ийг дотоод store-д хадгалж, idle TTL (анхдагч 15 минут) эхлүүлнэ. Дараа нь хэрэглэгч чат интерфейсээр асуулт асуухад session ID нь AI сервер рүү дамжина. AI сервер MCP хэрэгсэл дуудахдаа энэ session ID-г ашиглана. MCP сервер нь session хүчинтэй эсэхийг шалгаж, SISI API руу хүсэлт илгээнэ.

Энэ шийдлийн давуу талууд нь хэд хэдэн байна. Нэгдүгээрт, порталын access token болон cookie-г AI загварт шууд ил гаргахгүй бөгөөд зөвхөн MCP серверээр дамжуулан ашиглана. Хоёрдугаарт, session idle TTL ашиглан локал сессийн насыг хязгаарлаж, хэрэглэгч идэвхгүй байх үед сессийг автоматаар устгана. Гуравдугаарт, хөтчийн origin allowlist ашиглан зөвшөөрөгдсөн эх сурвалжаас л сесс бүртгэх боломжтой болгоно. Дөрөвдүгээрт, ижил механизмыг цаашдын багшийн портал болон бусад MCP интеграцын permission flow-д өргөтгөх боломжтой.

\section{MCP хэрэгслүүдийн загвар}

SISi Intelligent Agent-ын MCP хэрэгслүүдийг домэйнээр нь салгасан. Энэ нь AI загварт ``ямар зорилгод ямар хэрэгсэл ашиглах вэ'' гэдгийг тодорхой болгоход тустай. Одоогийн байдлаар найман read-only хэрэгсэл тодорхойлогдсон.

\texttt{get\_student\_overview} хэрэгсэл нь оюутны үндсэн мэдээлэл, зураг, идэвхтэй хичээлүүд, удирдлагын төлөв, мэдээ зэргийг буцаана. \texttt{get\_schedule} хэрэгсэл нь хичээлийн болон шалгалтын хуваарийг буцаана, course эсвэл exam гэж шүүх боломжтой. \texttt{get\_academic\_record} хэрэгсэл нь GPA, transcript, transcript статистик, curriculum, дээд англи хэлний мэдээлэл, хүсэлтийн төлөв зэргийг буцаана. \texttt{get\_financial\_overview} хэрэгсэл нь төлбөрийн үлдэгдэл, банкны дансны мэдээлэл, өмнөх болон сүүлийн үлдэгдлийг буцаана. \texttt{get\_notifications} хэрэгсэл нь оюутны мэдэгдэл болон аппын notification-уудыг буцаана. \texttt{get\_student\_resources} хэрэгсэл нь гарын авлага, баримт бичиг, төлбөрийн семестрүүд, оюутны картын мэдээлэл, тайлангийн төрлүүдийг буцаана. \texttt{browse\_top\_menus} хэрэгсэл нь их сургуулийн нэгжүүд, хичээлүүд, хуваарийн top menu өгөгдлийг буцаана. \texttt{get\_ejournal\_summary} хэрэгсэл нь e-journal-ийн бүрэлдэхүүн хэсгүүд, journal бичлэгүүд, үнэлгээний хуваарилалтыг буцаана.

Хэрэгсэл бүр нь optional параметрүүдтэй бөгөөд хэрэглэгч шаардлагатай өгөгдлийг сонгон авах боломжтой. Жишээлбэл, \texttt{get\_student\_overview} хэрэгсэл нь \texttt{includePhoto}, \texttt{includeCurrentCourses}, \texttt{includeNews} зэрэг параметрүүдтэй. Энэ нь шаардлагагүй өгөгдлийг татахгүй байх, хариу хугацааг богиносгоход тустай.

\section{Ирээдүйн өргөтгөлийн загвар}

Эцсийн зорилтот системд хэд хэдэн өргөтгөл нэмэхээр төлөвлөж байна.

Нэгдүгээрт, ерөнхий MCP өргөтгөлүүд нь календарь, баримт бичиг, экспорт, мэдэгдэл зэрэг төрөл бүрийн гаднын үйлчилгээг MCP-ээр холбох боломжийг олгоно. Жишээлбэл, хуваарийн өгөгдлийг Google Calendar руу шууд дамжуулах, дүнгийг CSV файл болгох зэрэг.

Хоёрдугаарт, багшийн порталын интеграц нь багшийн API-г MCP хэлбэрт оруулж, дүн оруулах зэрэг mutation workflow-г аюулгүй болгоно. Ялангуяа Excel файлаас олон оюутны дүн уншуулан системд оруулах ажлыг хөнгөвчлөх зорилготой.

Гуравдугаарт, mutation баталгаажуулалт нь эрсдэлтэй үйлдлүүдэд хэрэглэгчээс дахин баталгаа авах механизм юм. AI загвар mutation хэрэгсэл дуудахаас өмнө хэрэглэгчид ``Та энэ үйлдлийг хийхийг хүсэж байна уу?'' гэсэн асуулт харуулж, зөвшөөрөл авсны дараа л үйлдлийг гүйцэтгэнэ.

Эдгээрийг AI серверт хатуу кодлохын оронд тусдаа MCP сервер буюу MCP-compatible tool layer болгон оруулах нь архитектурын хувьд илүү оновчтой. Ингэснээр SISi Intelligent Agent нь тодорхой нэг үйлчилгээтэй хэт холбогдсон хэрэгсэл бус, олон төрлийн системийг уялдуулах ерөнхий зохицуулагч давхарга болж хөгжинө.
